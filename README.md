# thesis-final-pipeline

An end-to-end pipeline that takes a lecture recording video and produces:

- Structured **intermediate representation (IR)** of whiteboard/blackboard diagrams as they are drawn
- **Tactile graphics** (BANA-compliant SVG + 60×40 Braille pin-grid) for display on a Graphiti refreshable tactile device
- **Pin-level deltas** between consecutive board states, so only changed pins need to be updated on the display
- A human-readable **transcript** of diagram evolution over time

The system is designed to make hand-drawn lecture diagrams accessible to blind and low-vision students in real time.

## Pipeline stages

### Stage 0 — Ingest

Reads video metadata (duration, frame rate, resolution). Detects the board region of interest (ROI) automatically, or accepts a manual `(x, y, w, h)` rectangle via `config.ROI_MANUAL`.

### Stage 1 — CV temporal segmentation

Uses a Gaussian Mixture Model background subtractor to detect activity in the ROI. Consecutive idle frames (configurable via `STABILITY_WINDOW`) are interpreted as a pen-lift. At each pen-lift the current frame is saved as a keyframe and the detected marks (contours, Hough lines) are stored in an `ElementRegistry`. Marks are numbered with coloured overlays to produce **Set-of-Marks (SoM)** annotated keyframe images.

| Parameter | Default | Meaning |
|---|---|---|
| `STABILITY_WINDOW` | 15 frames | Consecutive idle frames = pen-lift (~0.5 s at 30 fps) |
| `ACTIVITY_THRESHOLD_HIGH` | 0.02 | ≥ 2% foreground pixels = active |
| `MIN_CONTOUR_AREA` | 3000 px² | Ignore tiny noise blobs |
| `MAX_MARKS` | 20 | Keep only the N largest marks per frame |

### Stage 2 — VLM analysis (Gemini)

Sends the SoM keyframes to Gemini 2.5 Flash in two passes:

- **Operation pass** — the whole video at once → ordered list of drawing operations (CREATION, ADDITION, HIGHLIGHTING, ERASURE, COMPLETE_ERASURE)
- **Snapshot pass** — one keyframe at a time → per-mark descriptions (text, shape, colour, connections), symbol meanings, groupings, board state summary

Both passes are cached to disk (`vlm_cache.json`) so re-runs skip the API calls.

### Stage 3 — Mealy machine

A finite-state machine with four states drives the emission of IR:

| State | Meaning |
|---|---|
| `B` | Blank — board is empty |
| `I` | Idle — board has content, no current activity |
| `O` | Operating — a drawing operation is in progress |
| `E` | Erasing — an erasure operation is in progress |

Inputs are VLM operation types (CREATION, ADDITION, …), pen-lift signals (Ω), and idle timeouts (τ). Outputs are:

| Output | Meaning |
|---|---|
| `S` | Emit a static IR snapshot of the current board state |
| `P` | Emit an operation IR entry |
| `SP` | Emit both |

### Stage 4 — Static IR generation

Merges CV geometry (ground truth for positions and bounding boxes) with VLM semantics (ground truth for text, shape type, connections) into a dict conforming to `static-schema.json`.

Node filtering follows a priority rule:

1. If the VLM says a mark is a `"node"` → always include
2. If the VLM says `"connection"` or `"annotation"` → always exclude
3. If the VLM has no opinion → fall back to CV shape type (exclude if arrow/line)

Node IDs (`n1`, `n2`, …) are assigned by reading-order position (top-to-bottom, left-to-right with a 50 px row bucket). Connection IDs are `c1`, `c2`, …

If no nodes can be identified the snapshot is silently skipped rather than emitting an invalid IR with an empty nodes array.

### Stage 5 — Operation IR

Builds one operation entry per P emission and assembles all entries into a single document conforming to `operation-schema.json`. The document includes a summary with counts per operation type.

### Stage 6 — Validation and semantic deltas

1. Validates every static snapshot against `static-schema.json`
2. Validates the operation document against `operation-schema.json`
3. Verifies operation summary counts match the operations array
4. Checks the Mealy state history for undefined transitions
5. Computes a **semantic delta** between every consecutive snapshot pair → `deltas/delta_NNNN.json`
6. Runs an RFC 6902 JSON-patch round-trip check: patch applied to snapshot N must exactly reproduce snapshot N+1

Semantic deltas report `added_nodes`, `removed_nodes`, `modified_nodes`, `added_connections`, `removed_connections`. Nodes are matched by ID; connections by `(source, target, direction)`.

> **Known limitation:** node IDs are reassigned by position at each snapshot independently. A new node appearing above an existing one shifts all IDs, producing false "modified" entries instead of "added + unchanged". Fix requires proximity-based persistent ID assignment across snapshots.

### Stage 7 — Tactile rendering

For each validated snapshot, runs the tactile sub-pipeline and computes a pin-level delta.

**Tactile sub-pipeline** (`tactile_rendering/tactile/`):

1. Loads and validates the static IR JSON
2. Runs Graphviz (`dot`) to compute a BANA-compliant layout — minimum node size 1 inch × 0.5 inch, 0.5-inch node separation
3. Assembles a **diagram SVG**: nodes drawn as their shape (rectangle, diamond, oval, …) with a bold numeric ID centred inside; directed edges with filled arrowheads; 1.5 mm stroke weight (BANA minimum)
4. Renders a **key SVG**: "Key:" title (BANA prohibits "Legend"), thick outer border, each entry has a print-text row and a braille-dot row below it; braille is rendered as explicit SVG `<circle>` elements at calculated coordinates — not Unicode font characters — to eliminate font-metric variability across embossers
5. Rasterises the diagram SVG at 300 DPI via CairoSVG and thresholds to a 60×40 binary pin-grid

**Pin-level delta** (computed by stage 7 after each render):

- Compares the new grid against the previous snapshot's grid (or a blank board for the first snapshot)
- `pin_delta.npy` — int8 array: `+1` = pin raised, `−1` = pin lowered, `0` = unchanged
- `pin_delta.png` — debug image: green = raised, red = lowered, grey = unchanged
- `pin_delta_meta.json` — pin counts and percentage changed; use to decide partial vs full refresh

### Stage 8 — Transcript

Generates a human-readable description of the diagram's evolution, written to `transcript.txt` and `transcript.json`.

## Output structure

```
outputs/
├── stage1_cache.json           # CV segment + registry data (avoids re-running CV)
├── vlm_cache.json              # VLM responses (avoids re-calling the API)
├── snapshot_cache.json         # intermediate snapshot data
├── keyframes/                  # SoM-annotated keyframe PNGs from stage 1
├── snapshots/
│   └── snapshot_NNNN.json      # validated static IR, one per S emission
├── operations/
│   └── operations.json         # single aggregated operation document
├── deltas/
│   └── delta_NNNN.json         # semantic delta between snapshot N and N+1
├── validation_report.json      # all schema/Mealy/count errors (or {is_valid: true})
├── tactile/
│   └── snapshot_NNNN/
│       ├── diagram.svg         # BANA-compliant tactile diagram
│       ├── key_page_0001.svg   # BANA key (embosser document)
│       ├── layout_debug.svg    # Graphviz layout with real coordinates (debug)
│       ├── tactile.json        # sidecar: legend table, braille, BANA metadata
│       ├── grid.npy            # 60×40 uint8 binary pin-grid
│       ├── grid.png            # debug visualisation of the pin-grid
│       ├── pin_delta.npy       # int8 diff vs previous snapshot
│       ├── pin_delta.png       # debug: green=raised, red=lowered, grey=unchanged
│       └── pin_delta_meta.json # pin counts; use to decide partial vs full refresh
├── transcript.txt
└── transcript.json
```

## IR schemas

### Static IR — one board state

```json
{
  "id": "2026-04-21T15:51:55Z",
  "elements": {
    "nodes": [
      {
        "id": "n1",
        "shape": "rectangle",
        "text": "Computer",
        "position": { "x": 499, "y": 333 },
        "visual": { "color": "red" }
      }
    ],
    "connections": [
      {
        "id": "c1",
        "source": "n1",
        "target": "n2",
        "direction": "forward",
        "line_type": "solid"
      }
    ]
  },
  "structure": {},
  "state": { "board_state": "A rectangle labelled Computer connected to …" },
  "semantics": {},
  "provenance": {
    "model": "gemini-2.5-flash + opencv-cv",
    "confidence": "high"
  }
}
```

### Operation IR — whole-video operation log

```json
{
  "analysis": {
    "operations": [
      {
        "operation_id": 1,
        "operation_type": "CREATION",
        "timestamp_start": "00:00",
        "timestamp_end": "00:15",
        "description": "Drew a rectangle labelled Computer"
      }
    ],
    "summary": {
      "creation_count": 1,
      "addition_count": 3,
      "highlighting_count": 0,
      "erasure_count": 0,
      "complete_erasure_count": 0
    }
  }
}
```
